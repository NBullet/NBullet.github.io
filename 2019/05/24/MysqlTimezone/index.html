MyBatis的时区问题
发布于 2019-05-24   更新于 2019-05-27 技巧 mysql jdbc

概述

在项目中发现date类型的数据，在提交后出现了数据不一致，经过对比，发现大概相差14个小时，第一时间便想到了时区问题

临时解决办法

在连接url中增加时区设置

经过设置后，数据查询、保存都不再有问题

排查问题

1、检查MySql时区设置是否正确？

既然是时区的问题，我们首先想到的就是检查MySql设置的时区是否正确：

通过命令 show variables like ‘%time_zone%’ 可以查询到当前的时区是CST

2、CST时区具体是指什么呢？

CST居然可以表示多个时区，而美国中部时间和中国标准时间刚好相差14个小时，与我们数据库里面的误差时间刚好相同。

如果说Mysql中的CST表示的就是中国标准时间，那么Mysql的设置确实没有问题

3、Mybatis认识CST和Mysql认识的CST不同

在网上找到已经有人分析过类似的问题，引用一下:https://blog.csdn.net/yjgithub/article/details/80404002

现在开始调试mybatis源码，调试到mysql-connector-java-6.0.4.jar包的com.mysql.cj.jdbc.io.JdbcTimestampValueFactory的createFromTimestamp方法时，发现mysql的底层驱动程序对从数据库查询出来的时间用了一个Calender做类型转换，Calender记录中包含的时区为CST，跟中国的时区Asia/Shanghai正好差了14小时。

那么为什么mybatis连接数据库会使用CST的美国时间呢？继续查看源码发现

mysql连接数据库的时候会从mysql读取系统的时区设置，调试com.mysql.cj.mysqla.MysqlaSession.java的configureTimezone方法发现，this.getServerVariable(“system_time_zone”)从系统里面读出来的时区设置是CST

从文中我们了解到:

  • Mysql中设置的时区来之操作系统
  • Mysql中的CST表示中国标准时间
  • 文中提到的Mybatis的调用实际调用的是jdbc的内容，并且jdbc中的CST表示美国中部时间

解决办法

从上面的分析中，我们找到2个解决办法

1、用jdbc可以识别的定义来重新设置mysql的时区：

2、修改mysql的配置文件，在修改 my.cnf 文件,在 [mysqld] 节下增加 default-time-zone = '+08:00'

3、如上面的临时解决办法，在连接方式中指定时区
